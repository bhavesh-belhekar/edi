# AI-Driven SOC / SIEM Research Environment

## Local Development & Containerization

Welcome to the collaborative setup for the **Synthetic Telemetry Generation** pipeline. To eliminate "it works on my machine" issues and mismatched dependencies among the 5 team members, we have containerized the baseline generator subsystem using Docker.

### 🏗 Architecture Notes
- **Simple & Clean:** Only the generation, streaming, and validation elements are containerized here.
- **Modularity:** Databases (Elastic/Wazuh/Ollama) are excluded *for now*. The `docker-compose.yml` serves as the foundational skeleton to inject them in upcoming phases.
- **Dataset Separation:** NDJSON files generated inside the container are strictly divided by source (`auth`, `endpoint`, `dns`, `firewall`, `proxy`) and automatically persist on your host machine in `synthetic_logs/output/` via Volume mounts.

---

### 🚀 Getting Started

#### 1. Build the Development Environment
Run this once to build the `python:3.11-slim` image and install the libraries defined in `requirements.txt`:
```bash
docker compose build
```

*(Note: We mount your current working directory to `/app` directly into the container. Any edits you make to `.py` scripts locally will reflect immediately without needing a rebuild!)*

---

### 💻 Common Workflows

We use `docker compose run` to fire ephemeral containers that execute a task and immediately exit.

#### 1. Generate Telemetry Logs
To execute the log generator and spawn coordinated datasets:
```bash
docker compose run --rm synthetic-logs python synthetic_logs/generate_logs.py --count 500 --attacks 3
```
*Outputs will populate natively on your host machine inside `synthetic_logs/output/`.*

#### 2. Validate Generated Logs
After generating telemetry, run the complete validation script:
```bash
# Assuming validate_output.py or validate.py depending on your flow
docker compose run --rm synthetic-logs python synthetic_logs/validate_output.py
```

#### 3. Stream Telemetry
*(For future SIEM/Kafka ingestion pipelines)*
```bash
docker compose run --rm synthetic-logs python synthetic_logs/stream_logs.py
```

#### 4. Interactive Bash Shell
If you need to root around inside the container environment to debug paths, check dependencies, or test scripts interactively:
```bash
docker compose run --rm synthetic-logs /bin/bash
```

### 📂 How Volumes Persist Data
The `docker-compose.yml` mounts:
- Host `./` ➔ Container `/app`
- Host `./synthetic_logs/output` ➔ Container `/app/synthetic_logs/output`

**Impact:** Everything generated by the orchestrator inside Docker bypasses the container's isolated filesystem and drops safely onto your native Host OS drive. These logs are preserved when the container spins down.

### 🔮 Future Scaling (Roadmap)
This repository is architected modularly. In subsequent project phases, you will seamlessly append new services to the `docker-compose.yml`:
- `wazuh-manager`: For direct native log ingestion.
- `ai-analysis`: A microservice polling logs and feeding vectors to an LLM.
- `vector-db`: Qdrant or Milvus for embedding long-term correlation mappings.
